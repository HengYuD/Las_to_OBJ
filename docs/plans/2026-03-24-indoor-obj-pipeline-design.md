# 室内 LAS 到 OBJ 结构模型流水线设计

## 需求摘要

### 功能需求

- 输入 OmniSLAM R8 采集得到的室内 `LAS` 点云
- 支持按 `ROI` 剔除区域外点云
- 过滤游离点和扫描噪声
- 识别墙面、地板、天花板等主结构平面
- 对结构平面做打薄和平整
- 输出适合射线追踪仿真的轻量 `OBJ`
- 生成中间调试产物，便于回看每一步效果

### 非功能需求

- 单层 `1.8G` 级别点云可分块读入
- 输出模型优先控制 mesh 数量和体积，而不是追求几何细节
- 算法必须可参数化，方便针对不同楼层迭代调参
- 流水线要可批处理、可重跑、可扩展

### 约束

- 初期不做云端系统，先做本地离线批处理
- 初期不解决家具、门窗洞口、复杂曲面和 BIM 语义
- 优先建立工程骨架和端到端链路，再逐步提高重建质量

## 候选方案

### 方案 A：Python 单体离线流水线

`laspy + numpy + open3d`，通过配置文件驱动分阶段处理。

优点：

- 落地快，适合从零搭仓库
- 算法试验和参数迭代成本低
- 后续可平滑拆出 GUI、批处理器和服务化接口

缺点：

- 对超大点云和复杂几何的性能天花板不如 C++
- 复杂布尔重建和拓扑修复能力有限

### 方案 B：CloudCompare / PDAL / Meshlab 工具链拼接

优点：

- 可快速验证单步算法效果
- 适合人工探索参数

缺点：

- 难形成稳定可复现系统
- 自动化和批量处理差

### 方案 C：C++ 核心库 + Python 封装

优点：

- 长期性能最好

缺点：

- 对当前阶段明显过重

### 结论

首版采用方案 A。先用 Python 把工程结构、数据模型、配置系统、调试产物和最小可用算法链路建立起来。等实际数据和参数稳定后，再决定是否把热点阶段替换成 PDAL 或 C++。

## 高层架构

```text
LAS/LAZ
  -> 分块读取
  -> ROI 裁剪
  -> 本地点云坐标系归一
  -> 体素降采样
  -> 统计离群点过滤
  -> RANSAC 平面提取
  -> 平面分类(墙/地/顶)
  -> 平面打薄与矩形 patch 建模
  -> OBJ 导出
  -> 报告与调试点云输出
```

## 关键决策

1. 首版只输出围护结构，不保留家具和细节。
2. 首版 mesh 使用矩形 patch，而不是复杂多边形重建。
3. 在 LAS 读取阶段就执行 ROI 过滤，减少后续内存压力。
4. 输出 OBJ 前统一平移到局部坐标系，避免大地坐标导致仿真精度问题。

## 失败模式与缓解

| 失败模式 | 影响 | 缓解 |
| --- | --- | --- |
| ROI 过大导致内存压力过高 | 处理失败或过慢 | 分块读取 + 先裁剪 |
| 家具点云干扰墙面提取 | 结构面破碎 | 增大离群过滤和最小平面点数 |
| 同一墙面被分裂成多个 patch | mesh 数量偏多 | 后续加入共面融合 |
| 门洞被错误补平 | 仿真几何失真 | 后续升级为轮廓重建和洞口识别 |
| 坐标过大导致 OBJ 精度问题 | 仿真异常 | 输出局部坐标系 |

## 演进路线

### Phase 1

- 建立仓库与 CLI
- 跑通端到端结构面 OBJ 导出

### Phase 2

- 交互式可视化 ROI 框选
- 共面墙体融合
- 平面边界清理

### Phase 3

- 门窗洞口识别
- 多边形轮廓建模
- 楼层批处理与缓存
